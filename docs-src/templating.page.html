Kempo Server includes a built-in XML-based templating system for building static sites with shared layouts, reusable fragments, and dynamic content. Pages can be pre-rendered at build time or server-side rendered on each request.

Overview

The templating system uses four file types that work together:

  • Templates (*.template.html) — Shared page layouts with named content slots
  • Pages (*.page.html) — Individual pages that fill template slots with content
  • Fragments (*.fragment.html) — Reusable HTML partials included in templates or other fragments
  • Globals (*.global.html) — Site-wide content blocks automatically injected into every page render

All three file types are blocked from being served directly by the default disallowedRegex configuration.

File Types

A typical project structure:

my-site/
├─ default.template.html    # Shared layout
├─ nav.fragment.html        # Reusable navigation
├─ footer.fragment.html     # Reusable footer
├─ analytics.global.html    # Site-wide scripts injected into every page
├─ index.page.html          # Homepage → renders to index.html
├─ about.page.html          # About → renders to about.html
├─ blog/
│  ├─ index.page.html       # Blog index → blog/index.html
│  ├─ post-1.page.html      # Blog post → blog/post-1.html

Templates and fragments are resolved by walking up the directory tree from the page file to the root, so subdirectories can override them by providing their own versions.

Templates

A template defines the shared HTML structure for your pages. Use <location> tags to define named content slots that pages will fill.

<!-- default.template.html -->
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8" />
  <title>{{title}}</title>
</head>
<body>
  <fragment name="nav" />
  <main>
    <h1>{{pageName}}</h1>
    <location />
  </main>
  <location name="scripts">
    <!-- default scripts if page doesn't provide any -->
  </location>
</body>
</html>

Location Tags

Locations are named slots in a template that pages fill with content. The name attribute is optional — a <location> without a name defaults to "default".

  • <location /> — Self-closing default slot (equivalent to <location name="default" />)
  • <location name="scripts" /> — Named slot, renders empty if the page doesn't provide content for it
  • <location name="scripts">...fallback...</location> — Includes fallback content used when the page doesn't override it

Pages

A page file wraps its content in a <page> root element and uses <content> blocks to fill the template's locations. The location attribute is optional — a <content> block without one targets the "default" location. Multiple <content> blocks targeting the same location are merged and sorted by priority (higher first, default 0).

<!-- about.page.html → renders to about.html -->
<page pageName="About Us" title="About - My Site">
  <content>
    <p>Welcome to our about page.</p>
  </content>
</page>

Page Attributes

Attributes on the <page> tag become template variables. In the example above, {{pageName}} resolves to About Us and {{title}} resolves to About - My Site.

The special template attribute selects which template to use. It defaults to default, which looks for default.template.html.

<!-- Uses blog.template.html instead of default.template.html -->
<page template="blog" title="My Post">
  <content location="main">...</content>
</page>

Frontmatter

Any content before the opening <page> tag is ignored by the renderer. This makes it a natural place for author metadata, description, dates, or any notes that belong with the file but shouldn't appear in the output — similar to frontmatter in other static site generators.

<!--
  author: Dustin Poissant
  date: 2026-04-10
  description: Overview of the routing system
-->
<page title="Routing">
  ...
</page>

The comment is purely for the author — nothing before <page> is parsed or rendered.

Fragments are reusable HTML partials. Include them in templates or other fragments using the <fragment> tag.

<!-- nav.fragment.html -->
<fragment>
  <nav>
    <a href="./">Home</a>
    <a href="./about.html">About</a>
  </nav>
</fragment>

Include it in a template:

<!-- Self-closing: renders empty if fragment file not found -->
<fragment name="nav" />

<!-- With fallback content if fragment file not found -->
<fragment name="nav"><nav>Fallback Nav</nav></fragment>

Fragments can include other fragments. The maximum nesting depth is controlled by maxFragmentDepth (default: 10).

File Resolution

When a page references a template or a template includes a fragment, the system searches for the file starting in the page's directory and walking up to the root. This means:

  • Templates and fragments placed at the root apply to all pages
  • A subdirectory can provide its own version to override the parent

Global Files

Any file named *.global.html anywhere under the root directory is automatically loaded and its <content> blocks are injected into every page render. This is the right place to put site-wide scripts, meta tags, nav links, or anything else that should appear across all pages without touching every template or page file.

<!-- analytics.global.html -->
<content location="scripts">
  <script src="analytics.js" defer></script>
</content>

The template just needs a matching <location> tag — the global file fills it automatically on every page.

Priority

When multiple <content> blocks — from global files, multiple global files, or the page itself — target the same location, a priority attribute controls their output order. Higher numbers come first. The default is 0.

<!-- framework.global.html -->
<content location="scripts" priority="10">
  <script src="framework.js"></script>
</content>

<!-- analytics.global.html — priority defaults to 0 -->
<content location="scripts">
  <script src="analytics.js" defer></script>
</content>

Output: framework.js first (priority 10), then analytics.js (priority 0). The same priority attribute works on <content> blocks in page files too, so a page can insert content before or after global content in the same location.

Locations Inside Page Content

You can place <location> tags inside a page's <content> block. Global content fills those locations before the block is inserted into the template. This lets a page pull in a global snippet at a specific spot without knowing what the snippet contains.

<!-- nav-links.global.html -->
<content location="links">
  <a href="./about.html">About</a>
</content>

<!-- nav.fragment.html — location tag pulls in the global links -->
<fragment>
  <nav>
    <location name="links" />
  </nav>
</fragment>

Fragments vs. Global Files

Fragments and global files both let you share HTML across pages, but they work in opposite directions.

Fragments are a pull. The template (or another fragment) explicitly asks for one thing by name: <fragment name="nav" />. The fragment file is a single reusable block of markup. Only what is asked for gets included, and where it lands is decided entirely by the caller.

Global files are a push. A *.global.html file declares content and targets a location, and that content is automatically injected into every page render without the template needing to know it exists. You can have many global files each contributing to the same location, and a page can add its own content to that same location alongside them. The final output is all of those contributions merged and ordered by priority.

FragmentsGlobal Files
DirectionPull — caller decides what to includePush — file injects itself automatically
How many per locationOne (the named file)Many — all globals targeting the same location are merged
Page can contribute too?No — the fragment replaces the tag entirelyYes — page <content> blocks merge with global content
Requires template change?Yes — caller must add <fragment name="..." />No — just add a <location> and the globals fill it
Good forReusable structural blocks (nav, footer, sidebar)Site-wide injections (scripts, meta tags, nav links)

Variables

Use {{variableName}} syntax to insert dynamic values into templates and fragments.

Built-in Variables

Variable Description
{{pathToRoot}}Relative path from the page to the root directory (e.g. ./, ../, ../../)
{{year}}Current four-digit year
{{date}}Current date in ISO format (YYYY-MM-DD)
{{datetime}}Full ISO 8601 datetime string
{{timestamp}}Unix timestamp in milliseconds
{{version}}Version from the root package.json
{{env}}Value of NODE_ENV

Page Attributes as Variables

Any attribute on the <page> tag is available as a variable in the template:

<!-- page file -->
<page title="My Page" author="Dustin">...</page>

<!-- template file -->
<title>{{title}}</title>
<meta name="author" content="{{author}}" />

Globals and State

Additional variables can be provided through the globals and state configuration objects. Globals and state are merged with page attributes, with page attributes taking priority.

// .config.js
export default {
  templating: {
    globals: {
      siteName: 'My Site',
      copyright: '© 2026 My Company'
    },
    state: {
      buildId: () => Date.now().toString(36)
    }
  }
};

Function values in globals or state are called at render time, allowing dynamic values.

Dot-Path Access

Variables support dot notation for nested object access:

{{site.name}}
{{author.email}}

Conditionals

Use <if> blocks to conditionally include content based on variable values.

<if condition="showBanner">
  <div class="banner">Welcome!</div>
</if>

Supported Operators

Conditions support a full expression syntax:

Operator Description
===Strict equality
!==Strict inequality
>, <, >=, <=Comparison
&&Logical AND
||Logical OR
!Logical NOT
( )Grouping

Examples

<!-- Truthy check -->
<if condition="isLoggedIn">Welcome back!</if>

<!-- Negation -->
<if condition="!isLoggedIn">Please log in.</if>

<!-- String comparison -->
<if condition="env === 'production'">
  <script src="analytics.js"></script>
</if>

<!-- Compound conditions -->
<if condition="isAdmin && hasPermission">
  <a href="/admin">Admin Panel</a>
</if>

Loops

Use <foreach> blocks to iterate over arrays.

<foreach in="navLinks" as="link">
  <a href="{{link.url}}">{{link.label}}</a>
</foreach>

The in attribute references an array variable and as names the loop variable. The loop variable supports dot-path access for object items.

Provide the array through globals or state:

// .config.js
export default {
  templating: {
    globals: {
      navLinks: [
        { url: '/', label: 'Home' },
        { url: '/about.html', label: 'About' }
      ]
    }
  }
};

Rendering

Pages can be pre-rendered to static HTML files using the CLI tool or the programmatic API.

CLI

# Render to a separate output directory
npx kempo-server-render ./src ./dist

# Render in-place (output to same directory)
npx kempo-server-render ./src

# With a state file
npx kempo-server-render ./src ./dist state.json

The CLI automatically loads .config.js or .config.json from the input directory for globals, state, and maxFragmentDepth settings.

Programmatic API

import { renderDir, renderPage } from 'kempo-server/templating';

// Render all pages in a directory
const count = await renderDir('./src', './dist', globals, state, maxDepth);

// Render a single page
const html = await renderPage('./src/about.page.html', './src', globals, state, maxDepth);

Server-Side Rendering

Pages can also be rendered on each request instead of pre-built. This is useful during development so you can see changes without rebuilding.

SSR Fallback

When ssr is enabled, the server first looks for a static file. If no static file is found, it looks for a matching .page.html file and renders it on the fly.

// .config.js
export default {
  templating: {
    ssr: true
  }
};

SSR Priority

When ssrPriority is also enabled, the server renders the .page.html file before checking for static files. This ensures you always see the latest template changes, even if a pre-rendered .html file exists in the same directory.

// dev.config.js
export default {
  templating: {
    ssr: true,
    ssrPriority: true
  }
};

This is ideal for development. For production, pre-render your pages and serve them as static files with SSR disabled.

Configuration

All templating options live under the templating key in your configuration file.

Option Default Description
preRenderfalsePre-render pages on server start
ssrfalseEnable server-side rendering as a fallback
ssrPriorityfalseRender pages before checking for static files (requires ssr: true)
globals{}Variables available to all pages
state{}Additional variables (function values are called at render time)
maxFragmentDepth10Maximum fragment nesting depth

Development vs Production

A common pattern is to use separate config files:

// dev.config.js — live rendering, no rebuild needed
export default {
  templating: { ssr: true, ssrPriority: true }
};

// prod.config.js — static files only
export default {
  templating: { ssr: false }
};
# Development
node dist/index.js -r ./docs/src -c dev.config.js

# Production (serve pre-rendered output)
node dist/index.js -r ./docs/dist
